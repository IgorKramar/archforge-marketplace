# krait_arch

[English](./README.md) | **Русский**

Архитектурный инструментарий для Claude Code. Архитектор уровня staff/principal-инженера на расстоянии одной команды, организованный вокруг повторяемого цикла и набора узких специализированных скиллов.

> **Философия.** Архитектура — это не документ, а *цикл решений* и *след, который эти решения оставляют*. Этот плагин делает цикл первоклассной сущностью, а след — долговечным.

---

## Архитектурный цикл

```
            ┌──────────────┐
            │  1. DISCOVER │  Ограничения, силы, прецеденты, требования
            └──────┬───────┘
                   ▼
            ┌──────────────┐
            │  2. DESIGN   │  2–3 альтернативы, у каждой — компромиссы
            └──────┬───────┘
                   ▼
            ┌──────────────┐
            │  3. DECIDE   │  Выбираем одну. Обосновываем. Указываем границы.
            └──────┬───────┘
                   ▼
            ┌──────────────┐
            │  4. DOCUMENT │  ADR + обновление ARCHITECTURE.md + диаграммы
            └──────┬───────┘
                   ▼
            ┌──────────────┐
            │  5. REVIEW   │  Архитектурное код-ревью на соответствие ADR
            └──────┬───────┘
                   │
                   └──── возвращается в DISCOVER для следующего изменения
```

У каждой фазы — своя slash-команда. Полный цикл по одной задаче запускается командой `/krait_arch:cycle "<постановка проблемы>"`.

Цикл повторяется. Каждый проход обогащает `docs/architecture/` — это и есть «компаунд»: следующее решение опирается на предыдущие, а не изобретается заново.

## Компоненты

### Slash-команды (`/krait_arch:*`)

| Команда | Назначение |
|---|---|
| `/krait_arch:init` | Подготовить проект — создать `ARCHITECTURE.md` и `docs/architecture/` |
| `/krait_arch:upgrade` | Мигрировать артефакты проекта на текущую установленную версию плагина |
| `/krait_arch:map` | Построить или обновить карту решений (группы, зависимости, рекомендуемый порядок) |
| `/krait_arch:observe` | Найти архитектурные пробелы — неявные решения в коде, устаревшие отложки, стратегия без архитектуры |
| `/krait_arch:discover <тема>` | Фаза 1 — собрать контекст |
| `/krait_arch:research <тема>` | Фаза 1.5 — собрать актуальную информацию из веба |
| `/krait_arch:design <тема>` | Фаза 2 — сгенерировать альтернативы |
| `/krait_arch:decide <тема>` | Фаза 3 — выбрать с обоснованием |
| `/krait_arch:document <тема>` | Фаза 4 — выпустить ADR и обновить корневой документ |
| `/krait_arch:review [путь]` | Фаза 5 — архитектурное код-ревью (с отслеживанием закрытия findings) |
| `/krait_arch:roast <ADR-NNNN\|путь> [--roles=...]` | Прожарка артефакта 5 ролями с разных перспектив |
| `/krait_arch:cycle <тема> [--scale=light\|standard\|deep]` | Прогнать весь цикл с детализацией под сложность задачи (auto-roast в deep) |
| `/krait_arch:adr <тема>` | Быстрый путь — сразу написать ADR |
| `/krait_arch:diagram <тип> <тема>` | Сгенерировать диаграмму: `c4-context\|c4-container\|c4-component\|sequence\|state\|er\|deployment` |
| `/krait_arch:c4 <уровень> <тема>` | Алиас для `/krait_arch:diagram c4-<уровень>` (для совместимости) |
| `/krait_arch:remember-compound-integration [--lang=en\|ru\|auto]` | Записать в `AGENTS.md` правила интеграции с плагином `compound-engineering` от EveryInc |

Все команды читают `ARCHITECTURE.md` и существующие ADR проекта как обязательный контекст.

### Скиллы

Маршрутизатор:

- **`architect`** — точка входа. Активирует архитектурную роль, решает, какие специализированные скиллы привлечь, поддерживает дисциплину цикла. Срабатывает автоматически, как только в разговоре появляются архитектурные темы (любое упоминание «архитектура», «дизайн», «масштабирование», «выбор стека», «компромиссы», «ревью архитектуры» и т.п.) — даже без slash-команды.

Специалисты (вызываются маршрутизатором или напрямую):

- **`c4-diagrams`** — модель C4: уровни context, container, component в Mermaid.
- **`adr-writing`** — Architecture Decision Records (шаблоны Nygard и MADR), жизненный цикл, антипаттерны.
- **`system-design`** — распределённые системы, масштабирование, БД, очереди, кэширование, CAP/PACELC, бюджеты задержек, монолит против микросервисов, разбор задачи в стиле системного интервью FAANG.
- **`frontend-architecture`** — стратегии рендеринга (CSR/SSR/SSG/ISR/RSC/Islands), матрица управления состоянием, структура модулей, производительность, edge-развёртывание.
- **`backend-architecture`** — декомпозиция сервисов, дизайн API, границы транзакций, асинхронные паттерны, наблюдаемость, отказоустойчивость.
- **`ai-agents-architecture`** — проектирование LLM-агентов, оркестрация инструментов, память, оценка качества, бюджеты задержки и стоимости, модель угроз prompt-injection.
- **`code-review-architectural`** — ревью со структурной перспективы (границы, связанность, согласованность, соответствие ADR), а не со стилистической.
- **`architecture-research`** — протокол получения свежей информации, когда утверждения зависят от версий, релизов, бенчмарков или текущих практик.
- **`compound-integration`** — описывает, как `krait_arch` чередуется с плагином `compound-engineering` от EveryInc: какая фаза куда вставляется, кто владеет каким артефактом, как избежать дублирования работы.

### Сабагенты

- **`architect`** — длительная архитектурная задача с глубоким разбором, возвращает структурированное предложение.
- **`reviewer`** — автономное архитектурное ревью директории или PR.
- **`researcher`** — фокусированное веб-исследование, возвращает дайджест с источниками.

Сабагенты полезны для задач, которые иначе засорят основной поток исследованиями или большими ревью.

### Хуки (мягкие предупреждения)

Плагин поставляется с **неблокирующими** хуками (`PostToolUse`), которые выдают напоминания, когда:

- Изменено много файлов без правки ADR (намёк, что архитектурное решение принимается неявно).
- Создан новый каталог верхнего уровня (намёк, что `ARCHITECTURE.md` стоит обновить).
- В пакетный манифест добавлена новая зависимость (намёк, что нужен ADR).

Хуки **никогда не блокируют** — только подталкивают. Архитектура — это дисциплина, а не шлагбаум.

### Шаблоны

- `templates/ARCHITECTURE.md` — корневой архитектурный документ. Живой, обновляется по мере эволюции системы.
- `templates/adr-template.md` — заготовка ADR.
- `templates/docs-architecture-readme.md` — индекс для `docs/architecture/`.

## Как работает `ARCHITECTURE.md`

Следуя [подходу Compound Engineering](https://wotai.co/blog/compound-engineering-agents-md), плагин разделяет проектную память на два файла:

- **`CLAUDE.md`** (он у тебя уже есть или появится) — контекст кодовой базы, конвенции, зависимости, раскладка файлов. *Как* выглядит код.
- **`ARCHITECTURE.md`** (этот плагин) — архитектурное состояние системы. *Почему* код выглядит именно так.

`ARCHITECTURE.md` — живой стержень. Содержит:

1. **Краткое описание системы** — что система делает, кто ею пользуется, с чем интегрируется.
2. **Атрибуты качества** — нефункциональные требования, формирующие дизайн (задержка, доступность, согласованность, масштаб).
3. **Высокоуровневая структура** — C4 уровни L1+L2 встроенно (Mermaid) или ссылки на `docs/architecture/diagrams/`.
4. **Ключевые ограничения** — организационные, операционные, регуляторные, бюджетные. То, что задаёт рамки пространства решений.
5. **Индекс решений** — ссылки на все ADR из `docs/architecture/decisions/` со статусом и одной строкой описания.
6. **Открытые вопросы** — известные неизвестные. Архитектурные вопросы, отложенные на будущие циклы.
7. **Антипаттерны проекта** — конкретные ловушки, которых команда договорилась избегать.

Claude читает `ARCHITECTURE.md` в начале сессии (через протокол скилла `architect`) и считает его обязательным контекстом. Каждое обновление ADR должно затрагивать и `ARCHITECTURE.md`, если меняется что-то из секций 1–6.

## Рекомендуемая раскладка в твоём проекте

После `/krait_arch:init` получится:

```
your-project/
├── ARCHITECTURE.md                    ← корневой архитектурный документ
├── CLAUDE.md                          ← (твоя проектная память)
└── docs/
    └── architecture/
        ├── README.md                  ← индекс этой директории
        ├── decisions/                 ← ADR (нумерованные, никогда не удаляются)
        │   ├── 0001-use-postgres-as-primary-store.md
        │   └── 0002-introduce-job-queue.md
        ├── diagrams/                  ← C4-диаграммы как .md (Mermaid) или картинки
        │   ├── context.md
        │   └── container.md
        ├── research/                  ← дайджесты исследований из /krait_arch:discover
        └── reviews/                   ← заметки архитектурного код-ревью
```

Можно отступить от этой раскладки — у каждой команды есть параметр `--root` или его аналог — но дефолты сделаны под неё.

## Тон и поведение

`krait_arch` имеет мнение. Маршрутизирующий скилл `architect` инструктирует Claude:

- Жёстко возражать против слабых предложений; держать позицию, пока не появится реальный контраргумент.
- Не давать рекомендаций по технологиям без предварительного выяснения ограничений и атрибутов качества.
- Всегда предлагать 2–3 альтернативы перед выбором одной.
- Явно указывать, при каких условиях рекомендация перестаёт работать (boundary conditions).
- Проверять версионно-чувствительные утверждения через свежий поиск в вебе перед фиксацией в ADR.

Это намеренно. Мягкие, угодливые архитектурные советы — самые дорогие: звучат полезно и тихо стоят годы рефакторинга.

Скилл намеренно «токсичный» в хорошем смысле — настойчивый, аргументирующий, не складывающийся при первом возражении. Это и отличает архитектурный диалог от вкусовой беседы.

## Языковая дисциплина

Скилл `architect` следует языку пользователя. Если общение по-русски — все артефакты (ADR, `ARCHITECTURE.md`, ревью, ответы в чате) генерируются на грамотном техническом русском, без чат-сленга.

В скилл встроен список устоявшихся русских технических терминов с заменой англицизмов-калек: «развёртывание» вместо «деплоймент», «наблюдаемость» вместо «обзервабилити», «отказоустойчивость» вместо «резильентность», «трассировка» вместо «трейсинг» и т.д. Имена собственные и общепринятые аббревиатуры (Postgres, gRPC, CDN, ACID, p95) остаются как есть.

Если предпочитаешь другие термины или хочешь дополнить список под свою команду — отредактируй секцию «Language and terminology» в `skills/architect/SKILL.md`.
